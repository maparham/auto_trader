# Multi-line MA Slope (+ smoothing) — design

**Date:** 2026-07-11
**Status:** Approved (brainstorming)
**Scope:** Frontend only
**Extends:** [[ma-slope-indicator]] (`docs/superpowers/specs/2026-07-10-ma-slope-indicator-design.md`)

## Summary

Generalize the just-shipped single-line `SLOPE` indicator to plot the slope of **up to
5 MAs** at once (one line per user-specified MA length) in a single sub-pane, add an
optional **smoothing** of the slope series, and make each line's style configurable. A
single length is simply the `N=1` case — this design **subsumes** the current single-line
behavior.

This reshapes the shipped SLOPE data model (length moves per-line into `calcParams`,
slope-period moves to config). No migration: nothing real persists a SLOPE rule (the only
instance was a removed verification test). Per [[no-legacy-code]], we change the shape
directly.

## User-facing behavior

- One **MA Slope** indicator; the settings modal has an **MA Lengths** editor (add/remove
  rows, **max 5**). Each length yields one slope line in the sub-pane.
- **Shared** across all lines: MA type (EMA/SMA), slope period N, units (%/hr | %/bar |
  price/bar), source, **smoothing**, and timeframe (MTF).
- **Per line**: the MA length, and the line's style (color / width / type).
- **Smoothing** (shared): None (default) / SMA / EMA of the slope series, with a
  configurable length. Applied after the slope (MA → slope → smoothing).
- **Color by direction** toggle: ON by default when there is **one** line — green when the
  slope is rising, red when falling (fixed green/red; configurable up/down direction colors
  are out of scope for v1). Automatically OFF and disabled when there are **2+** lines, where
  each line uses its own configured solid color (the configurable per-line styles). A single
  dashed zero reference line is always drawn.
- **Backtest/live rules** can reference, per length, BOTH the slope AND the underlying MA
  value as operands (e.g. "Slope MA 9" and "MA 9") — the MA is computed for rules but not
  newly plotted. See Rule operands.

## Data model

- **`calcParams` = the list of MA lengths** `[len0, len1, …]`, 1–5 entries (was
  `[maLen, slopePeriod]`).
- **`extendData` (`SlopeExtend`)**:
  - `slopePeriod: number` (moved here from calcParams[1])
  - `maType: "ema" | "sma"`
  - `units: "pctHr" | "pctBar" | "priceBar"`
  - `source` (MaExtend field)
  - `smoothing?: { type: "none" | "sma" | "ema"; length: number }`
  - `colorByDirection?: boolean` (direction mode uses fixed green/red)
  - `mtf?: { timeframe; htfStarts; htfSeriesByLine: Array<Array<number|undefined>>; htfMs }`
    — note `htfSeriesByLine` is **per-line** (was a single `htfSeries`).
- **Figures**: generated by a `regenerateFigures(calcParams)` callback → one line figure
  per length, keyed `slope0 … slopeK`. `calc` returns points `{ slope0, slope1, … }`.
- **Default styles**: `styles.lines` carries 5 default colors (a small palette) so the
  Style tab and draw have a per-line default for every possible line.

## Shared math (parity contract preserved)

The single shared module `frontend/src/lib/indicators/slope.ts` keeps ONE pipeline used by
the visual, the rule recipe, and the MTF coordinator:

```
slopeLineSeries(candles, maType, length, n, units, source, smoothing, barHours):
  base   = maSeries(candles, maType, length, {source}).base
  raw    = slopeWithUnits(base, n, barHours, units)      // existing helper, unchanged
  return smoothSeries(raw, smoothing)                    // none → raw unchanged
```

- `smoothSeries(values, {type,length})` applies an SMA/EMA over the slope array (reusing
  the array-based `sma`/`ema` helpers in `mtf.ts`; add an array `ema` if only `sma` exists).
- `barHours = inferBarHours(candles)` on whatever candle array each route holds — unchanged
  from [[ma-slope-indicator]], so %/hr agrees by construction.
- The multi-line `calc` maps `calcParams` (lengths) through `slopeLineSeries`, assembling
  `{ slope0, slope1, … }`. The rule recipe computes ONE line (`calcParams[line]`).

## Rendering & color

- `drawSlope` rewritten to iterate the active lines:
  - Reads each line's configured color from `params.indicator.styles?.lines[i]?.color`
    (fallback `params.defaultStyles.lines[i]` → default palette), plus size/dash.
  - If `colorByDirection` AND exactly one line → color each segment green (`≥0`) / red
    (`<0`) using the configured up/down colors.
  - Else → draw each line in its own configured solid color.
  - Always draws the dashed zero reference line once; guards `undefined` (warm-up) points.
  - Returns `true` to suppress the default figure lines.
- **Style tab**: already dynamic — it iterates `indicator.figures` and renders one
  color/width/type row per line (`IndicatorSettings.tsx` `lineDefs`/`setLine`/
  `lineOverrides`). N lines → N rows for free.

## Rule operands (slope AND underlying MA per length pickable)

Each length exposes **two** pickable operands: its **slope** and its **raw MA value** (the
MA is computed for rules but not newly drawn). For `K` lengths → `2K` operands.

- **`line` encoding** (relative to the recipe's own `calcParams`, `K = lengths.length`):
  `line ∈ [0, K)` → the slope of `lengths[line]`; `line ∈ [K, 2K)` → the raw MA of
  `lengths[line − K]`. `K` is read from the recipe's snapshotted `calcParams`, so it's
  stable regardless of later edits.
- `indicatorOutputs("SLOPE", …, calcParams)` (`chartOperand.ts`) returns `2K` outputs: for
  each length `i`, `{lineIndex: i, label: "Slope MA <len>"}` (line 0 = `base:true`) and
  `{lineIndex: K+i, label: "MA <len>"}`.
- `computeIndicatorRecipe` SLOPE case: `line < K` → `slopeLineSeries(candles, maType,
  lengths[line], n, units, source, smoothing, inferBarHours(candles))`; `line ≥ K` →
  the raw MA `maSeries(candles, maType, lengths[line−K], {source}).base` (no slope, no
  smoothing). Length stays **unclamped** (`Number(x)||default`) to match the visual.
- `operandBaseLen` SLOPE: slope operand (`line < K`) = `lengths[line] + slopePeriod +
  (smoothing enabled ? smoothing.length : 0)`; MA operand (`line ≥ K`) = `lengths[line−K]`
  (just the MA warm-up).
- **MTF for an MA operand** is automatic too: the rule path runs the recipe on native HTF
  candles regardless of whether it's a slope or a raw-MA line.
- Recipe hash: lengths (calcParams) + all shared params live in the hashed recipe
  (`extend` survives `sanitizeExtend`), so distinct configs get distinct `seriesKey`s.

## MTF (all lines)

- `applySlopeTimeframe` (`mtfCoordinator.ts`) computes `slopeLineSeries` for EVERY length on
  native HTF bars (`inferBarHours(htf)`), stashing `mtf.htfSeriesByLine` (array-of-arrays)
  before `alignHtfToChart`.
- The visual MTF branch in `computeSlopeCalc` aligns EACH line's HTF series onto chart bars
  and assembles `{ slope0, … }`.
- `refreshMtfIndicators` SLOPE branch reads the length list + shared params.
- Rule MTF is automatic: `buildChartOperandSeries` runs the recipe (one line) on native HTF
  candles, same as [[ma-slope-indicator]].

## Settings UI

`IndicatorSettings.tsx` (SLOPE branch), mirroring the existing per-type wiring:
- **MA Lengths** editor: up to 5 length inputs with add/remove (disabled add at 5). Writes
  `calcParams`. This is the one net-new control (the `indicatorMeta` fixed-input framework
  can't express a variable list) — a small dedicated component.
- Shared inputs: Slope Period, MA Type, Units, Source, **Smoothing** (type select + length),
  **Color by direction** toggle (auto-disabled when >1 length), Timeframe (MTF).
- All changes route through the SLOPE `applySlope`/apply path so an active MTF recomputes;
  persistence stores only `mtf:{timeframe}` (never the bulky HTF series), as today.

## Build risk to verify FIRST (spike task)

`regenerateFigures` is not used by any custom indicator here. `registerInstanceTemplate`
spreads `...base` (so the callback is preserved on the per-instance clone), but we must
confirm `overrideIndicator({ calcParams })` actually re-fires `regenerateFigures` and
re-emits the line figures on a **custom** per-instance SLOPE template. **Task 1 is a
runtime spike** proving this. **Fallback** if it doesn't fire: regenerate the instance's
figures manually and re-register + recreate the indicator when the length COUNT changes
(count changes are rare user edits; per-length value/color/param changes still use
`overrideIndicator`).

## Out of scope (v1)

- More than 5 lines.
- Per-line MA type / slope period / smoothing (shared only).
- Degrees/angle units, colored-MA-overlay variant (already out of scope in the base spec).
